### 任务7.1：实现流水线执行器（`pipeline/runner.py`）

- **任务题目**  
  实现主流水线执行器 `PipelineRunner`，按顺序调用 L1 → L2 → L3 → L4 → L5 → L6，支持配置启用/跳过，并将结果完整存入 `DataContext`。

- **基本内容**  
  编写 `src/pipeline/runner.py`，定义 `PipelineRunner` 类，提供 `run(data_context: DataContext) -> DataContext` 方法。该方法根据 `pipeline.yaml` 的配置，依次调用各层入口函数，并处理异常和日志。

- **任务实施的思路和结构**  
  1. 类 `PipelineRunner`：  
     - `__init__(self, config_path=None)`：加载全局配置和 `pipeline.yaml`。  
     - `run(self, data_context)` 方法：  
       依次执行 L1→L6，每个步骤前检查 `pipeline.stages.LX.enabled`。  
       每层调用对应的 `run_lX(context)` 函数（已在各层 `__init__.py` 中定义）。  
       每层执行后记录日志（`logger.info(f"Stage L{X} completed")`）。  
       若某层抛出异常，记录错误并根据配置决定中止还是继续（可配置 `fail_fast`）。  
       最终返回更新后的 `data_context`。  
  2. 支持只运行部分层（例如只运行到 L4 生成矛盾结果），通过配置 `stop_after` 参数实现。  
  3. 增加性能计时：记录每层耗时，存入 `context.metadata['timing']`。  
  4. 集成 `io_handler`（任务7.3）前，可先不处理输入输出，仅接收 `DataContext`。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`src/pipeline/runner.py`  
  - 参考：`具体核心项目结构V2.0.docx` 中 `runner.py` 描述；`技术路线总指南V2.0.docx` 整体架构数据流。

- **需要参考的具体论文或者模型**  
  无。

### 任务7.2：实现阶段管理器（`stage_manager.py`，支持动态跳过与依赖检查）

- **任务题目**  
  实现阶段管理器，管理各层之间的依赖关系，支持根据配置跳过层，并自动跳过依赖层。

- **基本内容**  
  编写 `src/pipeline/stage_manager.py`，提供 `StageManager` 类，维护各层名称、依赖关系（如 L3 依赖 L2）、启用状态，提供 `get_enabled_stages()` 和 `validate_dependencies()` 方法。

- **任务实施的思路和结构**  
  1. 定义 `STAGE_DEPENDENCIES` 字典：  
     ```python
     {
         "L1": [],
         "L2": ["L1"],
         "L3": ["L2"],
         "L4": ["L3"],
         "L5": ["L4"],
         "L6": ["L5"]  # L6 可使用前面所有层，但为简单，依赖 L5
     }
     ```  
  2. 在 `__init__` 中读取 `pipeline.yaml` 各层的 `enabled` 状态，并**合并 `input_profiles.yaml` 中当前 input_type 的 Profile**（覆盖 L3/L4/L6 开关与 L3 `segmentation_mode`）。  
  3. `validate()` 方法：若某层启用，但其依赖层未启用，则发出警告，并自动启用依赖层（或抛出异常，可配置）。  
  4. `get_execution_order()` 返回按依赖顺序排列的已启用层列表。  
  5. 供 `PipelineRunner` 使用。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`src/pipeline/stage_manager.py`  
  - 参考：`具体核心项目结构V2.0.docx` 中 `stage_manager.py` 提及。

- **需要参考的具体论文或者模型**  
  无。

### 任务7.3：实现输入输出处理器（`io_handler.py`）

- **任务题目**  
  实现输入输出处理器，负责加载原始视频/音频文件，初始化 `DataContext`，并在流水线运行后保存结果（JSON/MessagePack）及中间特征（可选缓存）。

- **基本内容**  
  编写 `src/pipeline/io_handler.py`，提供 `load_input(input_type, *, video_path=None, audio_path=None, text_content=None, image_path=None, text_subtype=None, user_id=None) -> DataContext` 和 `save_output(context, output_dir)` 函数。

- **任务实施的思路和结构**  
  1. `load_input`：  
     - 根据 `input_type` 校验必填字段（video→video_path；audio→audio_path；text→text_content 或 text_path；image→image_path）。  
     - 解析 `input_profiles.yaml` 得到 Profile，写入 `metadata.input_profile` 与 `metadata.active_modalities`。  
     - 使用 `audio_utils` / `video_utils` 提取基本信息（时长、采样率等），暂存到 `context.raw_data`。  
     - 初始化 `DataContext`，填充 `metadata`（会话ID、用户ID、时间戳）。  
  2. `save_output`：  
     - 将 `context` 序列化为字典（调用 `context.to_dict()`），保存为 JSON 文件（或 MessagePack 用于高效存储）。  
     - 可选：保存中间特征（如 L1 特征、L2 预测）到单独文件以节省空间（配置 `save_intermediate`）。  
     - 保存路径格式：`output_dir/{session_id}/context.json`，以及 `output_dir/{session_id}/figures/` 等。  
  3. 支持断点续跑：若已有缓存特征，可跳过 L1 直接加载（需实现特征缓存机制，可选后期）。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`src/pipeline/io_handler.py`  
  - 参考：`具体核心项目结构V2.0.docx` 中 `io_handler.py`。

- **需要参考的具体论文或者模型**  
  无。

- **预估时间（AI/agent辅助）**  
  1 小时

### 任务7.4：实现统一入口函数 `run_pipeline`

- **任务题目**  
  实现 `core` 库的统一入口函数 `run_pipeline(input_type, *, video_path=None, audio_path=None, text_content=None, image_path=None, text_subtype=None, user_id=None, config_overrides=None)`，返回包含完整结果的字典。

- **基本内容**  
  在 `core/__init__.py` 或 `pipeline/__init__.py` 中提供 `run_pipeline` 函数，封装 `io_handler`、`PipelineRunner` 和输出保存逻辑，对外暴露简洁接口。

- **任务实施的思路和结构**  
  1. 函数签名：  
     ```python
     def run_pipeline(
         input_type: str,
         *,
         video_path: Optional[str] = None,
         audio_path: Optional[str] = None,
         text_content: Optional[str] = None,
         image_path: Optional[str] = None,
         text_subtype: Optional[str] = None,
         user_id: Optional[str] = None,
         config_overrides: Optional[dict] = None,
         save_output: bool = True,
         output_dir: Optional[str] = None
     ) -> dict:
     ```  
  2. 内部流程：  
     - 加载配置（若提供 `config_overrides`，则合并）。  
     - 调用 `io_handler.load_input` 创建 `DataContext`。  
     - 实例化 `PipelineRunner`，调用 `runner.run(context)`。  
     - 若 `save_output` 为真，调用 `io_handler.save_output`。  
     - 返回 `context.to_dict()`（或结果子集，如报告、人格、矛盾等）。  
  3. 便捷函数：`run_pipeline_from_context(context)` 用于内部测试。  
  4. 错误处理：捕获异常，返回部分结果和错误信息。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`core/__init__.py`（或 `core/pipeline/__init__.py`）  
  - 参考：`具体核心项目结构V2.0.docx` 中 `core` 库的统一入口；`接口与web端相关项目结构V2.0.docx` 中 `run_pipeline` 描述。
