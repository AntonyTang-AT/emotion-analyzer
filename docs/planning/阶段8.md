### 任务8.1：定义 Pydantic Schemas（请求/响应模型）

- **任务题目**  
  定义所有 API 的 Pydantic 模型，包括用户、分析请求/响应、矛盾结果、片段结果、人格结果等，确保与 core 库的输出结构一致。

- **基本内容**  
  编写 `server/schemas/` 目录下的模块：`user.py`, `analyze.py`, `personality.py`, `common.py`，定义以下主要模型：  
  - `VAPredictionSelf` / `VAPredictionInter`（valence, arousal, confidence）  
  - `ContradictionResult`（type, intensity, involved_modalities, suggested_fusion_weights, routing_confidence）  
  - `SegmentResult`（id, start_time, end_time, va_self, va_inter, contradiction, report）  
  - `AnalysisResponse`（session_id, segments, overall_report, personality, raw_visual_summary）  
  - `UserCreate`, `UserResponse`, `BaselineRequest` 等  
  - `SimilarFragmentsRequest`（含 `embedding_type` 参数）

- **任务实施的思路和结构**  
  1. 使用 `pydantic.BaseModel`，启用 `orm_mode` 以便与数据库模型转换。  
  2. 嵌套模型：`SegmentResult` 内部包含 `va_self`（字典映射模态名到 `VAPredictionSelf`）和 `contradiction`。  
  3. `AnalysisResponse` 包含 `personality: Optional[PersonalityResult]`。  
  4. 所有字段与 core 库中 `DataContext` 的输出字典保持一致，便于直接序列化。  
  5. 添加字段验证（如权重列表长度为4，和为1等）。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`server/schemas/__init__.py`, `server/schemas/user.py`, `server/schemas/analyze.py`, `server/schemas/personality.py`, `server/schemas/common.py`  
  - 参考：`接口与web端相关项目结构V2.0.docx` 中 `server/schemas/analyze.py` 的代码示例；`技术路线总指南V2.0.docx` 中 L4、L6 输出结构。

### 任务8.3：实现向量存储封装（Chroma，支持两套 embedding）

- **任务题目**  
  在 `server/vector_store/` 中封装 Chroma 客户端，提供添加片段和查询相似片段的功能，支持 `embedding_type` 参数（self / inter）。

- **基本内容**  
  编写 `server/vector_store/chroma_client.py`（单例），`server/vector_store/memory_store.py`（业务接口）。  
  功能：  
  - `add_fragment(user_id, fragment_id, embedding_self, embedding_inter, metadata)`  
  - `query_similar(user_id, query_embedding, top_k, embedding_type, time_decay)`  
  - 支持时间衰减权重：`ω = exp(-α·Δt)`

- **任务实施的思路和结构**  
  1. 使用 `chromadb` 库，持久化目录 `data/chroma`。  
  2. 创建两个 collection：`fragments_self` 和 `fragments_inter`，或同一个 collection 存储两个向量字段（需自定义函数）。简便起见用两个 collection，但维护同一 `id`。  
  3. `add_fragment` 时，将 embedding_self 添加到 self collection，embedding_inter 添加到 inter collection，metadata 相同。  
  4. `query_similar` 根据 `embedding_type` 选择 collection，计算余弦相似度。若 `time_decay=True`，从 metadata 中读取 `timestamp`，计算衰减系数，乘以相似度后重排序。  
  5. 返回 `List[dict]` 包含片段元数据及相似度得分。  
  6. 提供清理过期数据的接口（可选）。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`server/vector_store/__init__.py`, `server/vector_store/chroma_client.py`, `server/vector_store/memory_store.py`  
  - 参考：`接口与web端相关项目结构V2.0.docx` 中 `vector_store/` 和 `memory_store.py` 修改说明；`技术路线总指南V2.0.docx` 中 L3.4 记忆模块。

- **需要参考的具体论文或者模型**  
  - Chroma 官方文档。

- **预估时间（AI/agent辅助）**  
  1.5 小时

### 任务8.4：实现用户认证与基线管理 API

- **任务题目**  
  实现用户注册、登录（JWT）、获取/更新基线（ΔVA）、冷启动相关接口。

- **基本内容**  
  编写 `server/api/v1/endpoints/user.py`，提供：  
  - `POST /register`：创建用户，存储密码哈希。  
  - `POST /login`：返回 JWT token。  
  - `GET /me`：获取当前用户信息。  
  - `POST /baseline`：上传基线校准结果（计算得到的 ΔVA）。  
  - `GET /baseline`：获取用户基线。  
  - `POST /cold_start`：基于首片段 `VA_self` 检索相似用户并返回临时 ΔVA（供前端/流水线使用）。

- **任务实施的思路和结构**  
  1. 使用 `python-jose` 处理 JWT，`passlib` 哈希密码。  
  2. 认证依赖：`deps.get_current_user` 解析 token 返回用户对象。  
  3. 基线 ΔVA 存储为 JSON 字符串或使用 `sqlalchemy.types.JSON`。  
  4. 冷启动接口：接收首片段的 `va_self` 向量（8维），调用 `memory_store.query_similar` 使用 `embedding_type='self'` 检索其他用户的片段，聚合其用户的 ΔVA（需要从 User 表获取）。加权融合返回临时 ΔVA。  
  5. 返回格式：`{"delta_va": [0.1, -0.2, ...], "source": "cold_start"}`。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`server/api/v1/endpoints/user.py`, `server/api/deps.py`, `server/utils/security.py`  
  - 参考：`接口与web端相关项目结构V2.0.docx` 中 `user.py` 和 `deps.py`。

- **需要参考的具体论文或者模型**  
  无。

- **预估时间（AI/agent辅助）**

### 任务8.5：实现多输入源上传与分析 API

- **任务题目**  
  实现 `POST /upload` 接口，接收 **video / audio / text / image** 输入，调用 `run_pipeline`，存储结果到数据库，返回 session_id。实现 `GET /result/{session_id}` 获取完整分析结果（含权重和两套VA，若 L4 启用）。

- **基本内容**  
  编写 `server/api/v1/endpoints/analyze.py`，提供：  
  - `POST /upload`：接收 `input_type`、对应 `file` 或 `text_content`、可选 `text_subtype` 与 `user_id`。后台任务执行分析（或同步，初期同步）。返回 `session_id`。  
  - `GET /result/{session_id}`：返回 `AnalysisResponse` JSON。  
  - `GET /report/{session_id}`：返回整体报告文本（从 result 中提取）。

- **任务实施的思路和结构**  
  1. 使用 `FastAPI` 的 `UploadFile` 接收文件，或 `Form` 接收 `text_content`；保存到临时目录（如 `data/uploads/`）。  
  2. 调用 `core.run_pipeline(input_type=..., ...)` 获取结果字典。  
  3. 将结果字典转换为 `AnalysisResponse` 模型（确保与 schema 一致）。  
  4. 创建 `Session` 记录，将 `result_json` 存入数据库。  
  5. 返回 `{"session_id": session_id}`。  
  6. 异步处理：可将分析任务放入 Celery 队列，但初期简单同步（对于短视频 <1分钟，可接受）。  
  7. 添加错误处理：视频格式不支持、分析失败等返回 4xx/5xx。  
  8. 实现 `GET /result` 时从数据库读取 JSON 并反序列化为 Pydantic 模型返回。

- **主要针对修改哪些具体的文件，需要参考哪些文件**

### 任务8.7：实现 WebSocket 实时流分析

- **任务题目**  
  实现 WebSocket 端点 `/ws/analyze`，接收视频帧（base64）或音频块，实时返回当前帧的 `VA_inter` 值，会话结束时返回完整报告和权重。

- **基本内容**  
  在 `server/api/v1/ws.py` 中实现 WebSocket 处理逻辑：  
  - 客户端每秒发送一帧（或音频块）。  
  - 服务端维护轻量级 L1+L2 预测，实时返回 `{valence, arousal, confidence, modal_contributions}`。  
  - 当客户端发送 `{"end": true}` 时，触发完整流水线（L3-L6），返回最终报告和权重时序图。

- **任务实施的思路和结构**  
  1. 使用 `fastapi.WebSocket`。  
  2. 维护会话状态：`session_id`, 累积的 `va_inter` 序列（每秒一个），以及原始特征缓存（可选）。  
  3. 对每个接收的帧：  
     - 解码 base64 图像，运行微表情/宏表情 L1（简化版），语音需要连续流（可仅用视频帧）。初期可只做宏表情+文本（用户不说话时）。完整实现需要流式语音处理复杂，建议初期只返回单帧的 VA_inter（基于当前帧的表情和最近文本）。  
     - 简化方案：只返回基于当前帧的宏表情 VA_inter 预测。  
  4. 客户端发送结束时，调用 `run_pipeline` 从累积数据生成完整结果（可能从头处理录制的视频，或使用缓存的特征）。  
  5. 将最终结果通过 WebSocket 发送（JSON 格式），并关闭连接。  
  6. 注意资源释放。

- **主要针对修改哪些具体的文件，需要参考哪些文件**  
  - 新建：`server/api/v1/ws.py`  
  - 参考：`接口与web端相关项目结构V2.0.docx` 中 `ws.py` 和 `camera.js` 提及。

- **需要参考的具体论文或者模型**  
  无特定，但流式处理可参考 WebSocket 标准实践。

- **预估时间（AI/agent辅助）**  
  2 小时（简化版）
